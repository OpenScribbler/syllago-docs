# Provider Pages Redesign — Design Document

**Status:** Draft — incremental. Sections marked **Decided** are settled; sections marked **Open** still need resolution before any plan or implementation work begins.

**Authors:** Holden Hewett + Maive
**Started:** 2026-04-14
**Supersedes:** `~/.claude/plans/lucky-gliding-globe.md` (Phase 2 plan that grew to ~90 hand-authored MDX files; abandoned in favor of the approach below)

---

## Problem

Provider pages today are generated by `scripts/sync-providers.ts`, which reads `src/data/capabilities/<provider>-<ct>.json` and writes large MDX files into `src/content/docs/using-syllago/providers/`. The current `claude-code.mdx` is over 21,000 tokens. The output has four concrete problems:

1. **Section titles read as catalogs, not conventions.** `### Kiro-specific Skills` looks like a list of skills, when it actually documents how Kiro implements the Skills content type.
2. **Per-item source links are visual noise.** Every `<dt>` wraps the same source URL 15-21 times within a section.
3. **No consolidated sources.** The `sources[]` list from each capability file isn't surfaced cleanly anywhere on the page.
4. **No depth.** The upstream `provider_extensions` shape (`id`, `name`, `description`, `source_ref`, `graduation_candidate`) carries only one description string per extension. There's no way to render "is this frontmatter field required?" or "what's the value type?" or show example syntax — because that information doesn't exist in the data.

The previous plan (see Supersedes) tried to solve #4 by hand-authoring ~90 MDX files in syllago-docs and adding a content collection for them. That introduced a content-duplication maintenance burden whose cost scales with upstream churn, and it didn't actually address #1-#3.

## Goals

A reader landing on a provider page can:

- See the provider's identity, detection method, and supported content types at a glance.
- Drill into how that provider implements each content type — sources, native format, canonical mappings, and provider-specific fields/behaviors.
- For each frontmatter-field extension, see at a glance whether it's required, what the value type is, and (where available) example syntax.
- For extensions where the upstream docs don't make required/optional clear, see an honest "unspecified" indicator instead of a misleading default.

The pages should render cleanly whether the upstream data is fully enriched or not. New depth flows in progressively as upstream YAMLs are enriched, with no flag day.

## Non-goals

- **Per-extension hand-authored MDX in syllago-docs.** Rejected; creates a parallel content store that drifts from upstream.
- **Side-channel YAML/JSON in syllago-docs for extension depth.** Rejected for the same reason.
- **Reproducing the previous plan's ~60 split per-CT pages** as the only option. Page granularity is still **Open** below; the choice is not pre-made.
- **Auto-classifying extensions into "frontmatter" vs. "topic" via regex on description text.** The upstream YAML will carry this distinction explicitly when needed.

## Decisions (as of 2026-04-14)

### D1. Push extension depth upstream into Syllago CLI

Three new fields land on `ProviderExtension` in the CLI repo's format-doc schema:

- `required` — nullable boolean. `true` = source explicitly says required. `false` = source explicitly says optional or shows a working example without the field. `null` (or unset) = source doesn't say; we don't know.
- `value_type` — short string describing the value shape (e.g., `string`, `string | string[]`, `bool`, `object`).
- `examples` — code blocks showing usage. (Exact structure still **Open** — see Q1.)

Source files: `/home/hhewett/.local/src/syllago/docs/provider-formats/<slug>.yaml` (14 files). Aggregated into `capabilities.json` by `cli/cmd/syllago/gencapabilities.go`. Synced into syllago-docs by the existing `scripts/sync-capabilities.ts`.

### D2. Render via Astro components, not generated MDX

`scripts/sync-providers.ts` stops writing per-provider MDX into `src/content/docs/using-syllago/providers/`. Provider pages become thin Astro/MDX files (or dynamic routes) that read from the `providers` and `capabilities` content collections and pass entries to components. Components own all layout decisions.

The exact component boundaries and page granularity are still **Open** (Q3, Q4).

### D3. Parallel rollout, no flag day

The work proceeds simultaneously in both repos:

- syllago-docs ships the new component-based pages first, rendering only what the current `capabilities.json` provides (name, description, source_ref). Pages look cleaner immediately.
- syllago/cli ships the schema bump (additive, optional fields). When YAMLs get enriched provider-by-provider, the next `sync-capabilities` run pulls the richer data, and the components automatically light up the new badges/types/examples.

This works because all new fields are optional in the Zod schema and rendered conditionally in the components.

### D4. Honest "unspecified" handling for `required`

When `required` is null:

- The component renders no badge (or a neutral visual cue — exact treatment **Open**, Q5), not a misleading "Optional" default.
- The Syllago CLI surfaces unspecified required-status through three coordinated channels (see D16): a `data_quality` block in `capabilities.json`, a footer note in `syllago info providers <slug>`, and a per-provider auto-managed GitHub tracking issue.
- Validation in `cli/internal/capmon/capyaml/validate.go` does **not** force `required` to be set — null is a valid value. (Other rules may still apply; see Q6.)

### D5. Enrichment uses existing capmon subagent pipeline

The YAMLs at `docs/provider-formats/*.yaml` already carry `generation_method: subagent` — they were initially populated by an LLM (Claude) reading the upstream docs. We extend that subagent's prompt to also extract the three new fields. No new LLM infrastructure.

The deterministic vs. LLM split:

| Field | Deterministic | LLM (subagent) |
|---|---|---|
| `required` | Regex on description for explicit "(required)" / "(optional)" markers (small fraction) | Reading prose in upstream docs to determine "you must specify X" / "X is mandatory" semantics (most cases) |
| `value_type` | Parsing YAML/JSON examples in docs to infer literal types via existing `extract_yaml.go` / `extract_json.go` | Resolving prose like "comma-separated list" → `string \| string[]` |
| `examples` | Code-block extraction by language fence (already works in capmon's extractors) | Curating which examples are representative vs. anti-pattern |

`capyaml/validate.go` enforces what we *can* verify: `value_type` strings must match an allow-list; `examples` code blocks must declare a language; etc.

### D6. Hybrid page granularity

Provider pages adopt a two-level structure:

- `/providers/<slug>/` — overview page (identity, detection, install methods, supported CT table). One per provider (~12 routes).
- `/providers/<slug>/<ct>/` — per-`(provider, contentType)` conventions page (sources, native format, canonical mappings, extensions). One per supported CT per provider (~60 routes total).

Sidebar nests one level: the provider name expands to "Overview" plus one entry per supported CT.

Trade-offs accepted:

- ~72 total routes (vs ~12 for single-page or ~60 for per-CT-only). Astro builds tolerate this fine.
- All 36 existing source-file links to `/using-syllago/providers/<slug>/` remain valid — the overview page *is* that URL.
- `src/components/CanonicalSupportTable.astro:67` continues to link to overview pages without modification.
- Page weight: claude-code's current ~21k tokens fragment into one small overview plus ~6 focused CT pages. No reader has to scroll past CTs they don't care about.

Why C over A (single page): even with cleaner components, the underlying capability data for claude-code keeps the page heavy. Splitting lets readers focus on one CT.

Why C over B (per-CT only): B deletes the top-level `/providers/<slug>/` URL, breaking 36 internal references and the table-of-providers link from canonical-key pages. The fix would be redirects or stub overview pages — and stub overview pages basically *is* option C, so we may as well make them real overviews.

### D7. Sources rendered as a top-of-page table

Each per-`(provider, contentType)` page leads with a single **Sources** table at the top, before any other section. Three columns:

| Source Name | Section it applies to | URL |
|---|---|---|

Rationale: source documentation is reference material — it belongs in one consolidated lookup, not scattered across section sub-bullets. The "Section" column gives provenance (which part of the page each source informed) without forcing the source list to repeat under every section. The previous handoff (`clarifying-provider-page-structure-source`) explicitly rejected source links embedded in section headings; this satisfies that constraint while improving on a flat bulleted list by adding section attribution.

### D8. Per-extension `source_ref` rolls into the Sources table

Each `provider_extensions[i].source_ref` URL becomes its own row in the Sources table, with `Section = "Extension: <field-name>"` (e.g., `Extension: model`). No inline "Learn more →" link inside the extension card — the table is the single home for every URL on the page.

Trade-off: the table grows with the extension count. For a CT with 20 extensions and 3 page-level sources, the table is 23 rows. Acceptable; tables scroll fine.

### D9. Sources schema gains `name` and `section` fields upstream

The `sources[]` schema in `provider-formats/<slug>.yaml` (and downstream in `capabilities.json`) gains two optional fields:

```yaml
sources:
  - uri: https://...
    type: documentation
    name: "Skills documentation"   # NEW — human-readable label
    section: "All"                  # NEW — which page section this informed
```

- Both fields are **optional** in the Zod schema (parallel rollout matches D1).
- Component fallbacks when a field is absent:
  - `name` → derived from the last path segment of `uri` (e.g., `skills.md` → `Skills`).
  - `section` → `"All"`.
- The `section` value for upstream sources matches one of: `"All"`, `"Native Format"`, `"Canonical Mappings"`, `"Extensions"` (or `"Extension: <field-name>"` only when generated for per-extension `source_ref` rows by the component, not authored upstream).
- The capmon subagent extends its prompt to populate `name` (always) and `section` (when the source is clearly tied to a single page section).

### D10. `examples` is a structured array

The `examples` field on `ProviderExtension` is an array of objects:

```yaml
examples:
  - title: "Set a fast model"        # optional, used as a caption
    lang: yaml                       # required, must match allow-list
    code: |                          # required, non-empty
      model: claude-haiku
    note: "Default if absent."       # optional, prose under the code block
```

Why structured over free-form markdown:

- The "author" of `examples` is the capmon subagent, not a human. Subagents are easier to constrain via schema than to coach via prose discipline. A schema with required fields produces predictable output; a "dump examples as markdown" prompt produces inconsistent output across providers.
- Validation can enforce per-example invariants (lang in allow-list, code non-empty, etc.) — see Q6.
- Component renders each item identically across providers, regardless of how the source docs phrase things.

Trade-off accepted: free-form markdown would let upstream docs preserve more nuance (e.g., "bad vs good" pairs, multi-paragraph context). The `note` field handles the most common case (a one-line caption); richer narrative can land later as a follow-up if needed.

### D11. Component split — discrete reusable sub-components

`ProviderConventions.astro` orchestrates a per-`(provider, contentType)` page but does not inline all section bodies. Sub-components:

- `SourcesTable.astro` — renders the top-of-page Sources table (D7-D9). Takes `sources[]` plus a list of per-extension `source_ref` rows. Reusable: documentation pages could later front-load a similar table.
- `ProviderCanonicalMappings.astro` — given `(provider, contentType)`, renders a table of canonical keys → mechanism for that specific provider. The perpendicular pivot of `CanonicalSupportTable.astro` (which pivots on canonical-key → providers).
- `ProviderExtension.astro` — one extension item: name + optional field-name code, optional Required/Optional badge, optional value-type, description body, optional examples (D10), optional source-row reference.
- `ProviderExtensionsList.astro` — wraps the iteration of `ProviderExtension`. Optional; can be inlined in `ProviderConventions` if the wrapping logic stays trivial.

`ProviderConventions.astro` itself reduces to: `SourcesTable` → inline "Native Format" prose → `ProviderCanonicalMappings` → `ProviderExtensionsList`.

`ProviderOverview.astro` (for `/providers/<slug>/`) is separate from `ProviderConventions` — it shows provider identity, detection, install methods, and a supported-CT summary table. It does *not* embed `ProviderConventions` (each per-CT page is the home of its conventions).

The existing `CanonicalSupportTable.astro` stays untouched and continues to render on `/reference/canonical-keys/<key>/` pages.

### D12. Three-state badge for `required`

The extension card displays a single badge representing the `required` field, with three distinct visual states:

| Value | Badge label | Style |
|---|---|---|
| `true` | **Required** | Filled, accent color (e.g., red/orange) |
| `false` | **Optional** | Outlined, neutral color |
| `null` (or absent) | **Unspecified** | Gray, dashed border |

All three badges occupy the same slot on the card, ensuring the page remains scannable. The "Unspecified" badge satisfies D4's "honest unspecified handling" by ensuring readers can never confuse "we know it's optional" with "the docs don't say."

Trade-off accepted: in the early rollout phase (before YAMLs are enriched provider-by-provider), most extensions land as "Unspecified". The page will look uniform during that period — that's an honest reflection of current data quality, not a UX bug.

### D13. Validation: strict allow-list with warn-don't-fail

`cli/internal/capmon/capyaml/validate.go` enforces shape-level rules (required fields present, types correct, non-empty values where applicable) as **errors** that fail the build.

For the new schema fields, an allow-list defines canonical values. Unknown values produce **warnings** logged at build time, but do not fail the build:

| Field | Allow-list (warn if outside) |
|---|---|
| `value_type` | `string`, `string[]`, `string \| string[]`, `bool`, `int`, `object`, `object[]`, `path` |
| `example.lang` | `yaml`, `json`, `toml`, `bash`, `javascript`, `typescript`, `python`, `markdown`, `mdx`, `ini`, `dotenv` |
| `sources[].section` | `All`, `Native Format`, `Canonical Mappings`, `Extensions` (extension-specific values like `Extension: <field-name>` are component-generated, not authored) |

Rationale: subagent enrichment (D5) generates novel values from time to time. Hard-failing CI on every novelty would slow the parallel rollout (D3) without commensurate benefit. Warnings surface drift to humans through D14's issue channel without blocking deploys.

### D14. Validation warnings open GitHub issues with full context

When `validate.go` emits a warning, the build flow opens (or updates) a GitHub issue containing the full context needed to triage:

- The YAML file path and line number where the warning was triggered.
- The field name and the value that produced the warning.
- The applicable allow-list and a brief explanation of why the value isn't on it.
- A suggested action (e.g., "add `array<string>` to the allow-list" or "rename to `string[]`").

Operational details (to be specified during plan phase):

- **Where issues land.** Likely the syllago/cli repo (since `validate.go` runs there), but the issue body should reference the docs build's URL when applicable.
- **Authentication.** CI runs use `GITHUB_TOKEN`. Local builds skip issue creation and log a one-line warning to stderr instead (no spamming the issue tracker from dev machines).
- **Deduplication.** Issues are keyed by `(file, field, value)` tuple. A repeated warning updates the existing issue (timestamp + occurrence count) rather than opening a new one.
- **Closing.** Issues auto-close when the underlying YAML is fixed and the warning no longer fires (build flow checks for the keyed issue and closes if absent on a clean run).

Rationale: warn-don't-fail (D13) keeps deploys moving, but warnings need to land in tracked work or they get lost. Auto-issued tickets ensure every uncertainty has an owner and an action.

### D15. No `kind` hint on extensions

`ProviderExtension` does not carry a `kind: frontmatter | topic | behavior` enum. Every field stays optional and validation is shape-based only.

The previous plan had a `kind` enum; we removed it deliberately and we're not bringing it back. Reasons:

- Adds a classification burden on the subagent (D5) for every extension, with no validation rule that strictly requires the distinction yet.
- Misclassification cascades into spurious validation failures or lost rendering hints.
- If a future need for kind-based behavior emerges, the field can be added additively without breaking the existing schema.

### D16. Data-quality channels for unspecified extension fields

Three coordinated surfaces expose the data-quality state of provider extensions (most importantly `required: null`):

1. **`capabilities.json` `data_quality` block (programmatic).** A top-level object aggregates per-provider counts and links:
    ```json
    {
      "data_quality": {
        "providers": {
          "claude-code": {
            "unspecified_required_count": 12,
            "unspecified_value_type_count": 8,
            "unspecified_examples_count": 17,
            "tracking_issue": "https://github.com/.../issues/123"
          }
        }
      },
      "providers": [...]
    }
    ```
2. **`syllago info providers <slug>` footer (human-CLI).** The CLI prints a one-line data-quality summary and the tracking-issue URL after the main provider info.
3. **Per-provider tracking issue (work-management).** A single auto-managed GitHub issue per provider lists every unspecified extension field, auto-updates each build (occurrence counts, latest seen), and auto-closes when all gaps are filled. Operationally this rides on D14's issue infrastructure (same auth, dedup logic, close-on-clean rules).

Rationale: machine-readable state for downstream consumers (including syllago-docs, which uses the counts to render summary banners), human-readable surface for CLI users, and tracked-work surface for maintainers — three different audiences, three appropriate channels.

`syllago doctor` is intentionally NOT used: doctor is for installation/configuration issues, not data-quality reporting.

### D17. Atomic single-PR migration

The docs-side migration ships in a single PR rather than a sequenced rollout. Contents:

- `scripts/sync-providers.ts` — keep the per-provider JSON-data writing (components need it); remove the MDX-generation portion (~600 lines).
- `src/content/docs/using-syllago/providers/` — delete the 13 generated MDX files (`claude-code.mdx`, `cursor.mdx`, etc.). Keep or rewrite `index.mdx` as the providers landing page.
- `src/pages/using-syllago/providers/` — add `[provider].astro` (uses `ProviderOverview`) and `[provider]/[ct].astro` (uses `ProviderConventions`). These dynamic routes replace the deleted MDX.
- `src/components/` — add `SourcesTable.astro`, `ProviderOverview.astro`, `ProviderConventions.astro`, `ProviderCanonicalMappings.astro`, `ProviderExtension.astro` (and optionally `ProviderExtensionsList.astro` per D11).
- `src/content.config.ts` — loosen the Zod schema per D9 (sources gain `name`, `section`) and D10 (examples becomes a structured array).
- `sidebar.ts` — replace flat provider entries with nested entries (Provider name → Overview + per-CT children).

Why atomic: deploying the schema, components, routes, and MDX deletion in lockstep avoids any in-between state where users see a mix of old MDX-rendered pages and new component-rendered pages. The PR is large but reviewable in one sitting; the alternative is multiple deploys with user-visible inconsistency between them.

URL preservation guarantees (verified during this design phase):

- All 36 source files referencing `/using-syllago/providers/<slug>/` continue to resolve (D6's overview pages are at those exact URLs).
- `src/components/CanonicalSupportTable.astro:67` continues to link to overview pages without modification.
- No anchor links to provider sub-sections (e.g., `#skills`, `#agents`) exist anywhere in the codebase, so no anchor-redirect work is needed.

Cross-repo coordination with syllago/cli (D3): the docs PR ships with the schema additions optional (so the components render gracefully against current `capabilities.json` immediately). The CLI repo's matching PRs (schema bump in `gencapabilities.go`, validate.go updates, `data_quality` block, `syllago info` footer) ride independently. Neither side blocks the other.

## Architecture sketch

```
syllago/docs/provider-formats/<slug>.yaml      ← source of truth (14 files, hand+subagent authored)
       ↓ cli/cmd/syllago/gencapabilities.go aggregates
capabilities.json                              ← GitHub release artifact
       ↓ syllago-docs/scripts/sync-capabilities.ts fetches + splits
syllago-docs/src/data/capabilities/*.json      ← per-provider, per-CT
       ↓ Astro content collection (existing, schema in src/content.config.ts)
       ↓ Astro components consume collection entries
syllago-docs/src/content/docs/using-syllago/providers/...   ← thin pages (or dynamic routes)
```

Loosened Zod schema in `src/content.config.ts`:

```ts
const providerExtensionSchema = z.object({
  id: z.string(),
  name: z.string(),
  description: z.string(),
  source_ref: z.string().url().optional(),
  // New optional fields, all rendered conditionally:
  required: z.boolean().nullable().optional(),
  value_type: z.string().optional(),
  examples: z.array(z.object({
    title: z.string().optional(),
    lang: z.string(),
    code: z.string().min(1),
    note: z.string().optional(),
  })).optional(),
});

// New optional fields on sources[] (D9):
const sourceSchema = z.object({
  uri: z.string(),                  // existing
  type: z.string().optional(),      // existing
  fetched_at: z.string().optional(),// existing
  name: z.string().optional(),      // NEW
  section: z.string().optional(),   // NEW
});
```
