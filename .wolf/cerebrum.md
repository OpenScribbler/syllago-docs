## User Preferences

- **AskUserQuestion previews must appear on ALL options when any option has one.** The user got confused mid-session when only the recommended option had a preview block — the partial display made it unclear what the other options actually meant. If you use `preview` on one option, populate it on every option in the question.
- **Never include time estimates** ("X minutes", "should take Y hours", "1-2 days") in plans, proposals, design docs, or task descriptions. Time framing assumes solo human pacing and doesn't apply to Holden+Maive collaboration. Use real cost dimensions instead: line counts, file count, blast radius, reversibility.
- **Pause before persistent shared-state actions** (creating beads, committing, pushing, opening PRs) — even when validation passes. The user explicitly wanted manual review of the implementation plan before 13+ beads entered the shared bd database. Default to surfacing the artifact + asking, not auto-proceeding.
- **Use AskUserQuestion for every question** — never plain-text numbered options. The clickable choice UI is materially easier to respond to than typing numbers.
- **Trade-offs in AskUserQuestion options must be substantive.** Don't list trivial downsides as if they were real costs (e.g., "an install script is more to maintain" when it's 20 lines and rarely changes). Recalibrate for the partnership: writing/maintaining code is fast and cheap; the real costs are reversibility, blast radius, and architectural lock-in.

## Key Learnings

- **Project:** syllago-docs
- **Description:** Documentation site for [syllago](https://github.com/OpenScribbler/syllago) -- the package manager for AI coding tool content.
- **`/develop` skill orchestrates feature workflows** in two phases: Phase 1 (brainstorm) is interactive and inline; Phase 2 (plan → quality review → validation → bead creation → execute) delegates each stage to subagents to keep the orchestrator's context lean. State lives in `.develop/<feature>.json`; hooks auto-update `currentStage`, `completed`, and `nextAction` after each stage's artifact is written.
- **Design docs go to `docs/plans/YYYY-MM-DD-<feature>-design.md`** and implementation plans to `docs/plans/YYYY-MM-DD-<feature>-implementation.md`. Reports go to `.develop/<feature>-quality-report.md` and `.develop/<feature>-validation-report.md`.
- **Validation is gated by a literal-string check on the report file.** The `create-beads.ts` hook reads `.develop/<feature>-validation-report.md` and looks for `✅ PASSED`. Manually setting `validation.passed = true` in the state file does NOT bypass the gate — the hook reads the report content directly. This is intentional: the orchestrator cannot rationalize past failed validation.
- **Handoffs use bd, not `.handoffs/<name>.json` files.** As of 2026-04-14, the legacy file-based handoff system was migrated. Handoffs are now bd issues with `label=handoff`, claimed via `assignee=session-<id>`. Title convention: `<what-was-done>-next-<what-the-next-session-does>` with `-next-` as the literal separator.
- **`bd ready --no-pager` does NOT work** in this version of bd (`unknown flag: --no-pager`). The handoff skill's workflow doc references this flag in multiple places but it must be omitted. `bd ready` and `bd list` already stream to stdout in non-TTY contexts.
- **Existing `CanonicalSupportTable.astro` pivots perpendicular** to per-provider rendering: it indexes (canonical-key, contentType) → list of providers. To render per-provider, per-CT views, a sibling component is needed (not reuse). This came up while resolving Q4 in the provider-pages-redesign brainstorm.
- **`scripts/sync-providers.ts` is dual-purpose** (1071 lines): it generates BOTH `src/data/providers/*.json` AND MDX documentation pages. The provider-pages-redesign D17 prunes only the MDX-generation half (~600 lines), keeping the data-writing half.
- **36 source files link to top-level `/using-syllago/providers/<slug>/` URLs.** Verified via grep early in the brainstorm. Any redesign that breaks top-level provider URLs needs to either preserve them (via overview pages) or sweep all 36 references — the cost of a flat per-CT-only restructure is not just architectural but link-debt.
- **`src/data/providers/<slug>.json` `content.<ct>.supported` is the authoritative filter** for the dynamic route, not the existence of `src/data/capabilities/<slug>-<ct>.json`. A capability JSON file can exist with `status: "supported"` while the provider JSON has `content.<ct>.supported: false` — and the route filter uses the provider JSON. Sidebar entries must match the route generation logic (provider JSON), not the capability file's presence. This caused the round-1 quality review's blocking sidebar issue (cline/skills, cline/commands, amp/hooks).
- **`src/content.config.ts` had uncommitted edits** for a now-superseded `provider-extensions` collection (from `scripts/seed-provider-extensions.ts`, also untracked). The provider-pages-redesign plan's Task 1 removes those edits and applies the design-doc schema changes (D9 sources name/section, D10 examples array, D12 nullable required) in their place. Don't pre-fix the dirty file — the plan handles the reset.
- **ProviderFeaturesTable partitions rows by `providerField !== null`**, not by "is canonical vs extension". A canonical mapping with `provider_field: null` lands in the "Other features" group (e.g., `canonical_filename` for skills: fixed filename rule, no native field). Canonical mappings linked via `extension_id` absorb the extension's `summary` + `conversion` — the extension is removed from the "other" group to avoid double-rendering. This is the key merge invariant; if a future contributor sees a canonical key showing up in the wrong section, the fix is almost always to populate `provider_field` upstream, not to change the partition logic.
- **Starlight `<h2 id="...">` headings must live OUTSIDE `not-content` wrappers** to register in the TOC sidebar. Sections wrapped in `class="not-content"` have Starlight's markdown prose styles disabled, and that also disables heading extraction for the right-rail TOC. In `ProviderConventions.astro`: At-a-glance is inside `not-content` because it's a grid, not prose — but its `<h2>` is rendered as a plain `<h2 class="at-a-glance__heading">`, deliberately NOT registered in TOC. Features and Sources h2s are outside `not-content` and DO appear in the TOC.
- **`.wolf/anatomy.md` entries can go stale when zombie files persist.** The 2026-04-14 migration removed `provider-extensions.css` in CHANGELOG but left the file on disk; anatomy.md still listed it. When doing cleanup passes, grep for the file across `src/` before trusting CHANGELOG removal claims — a zero-reference file is safe to delete. `astro.config.mjs` `customCss` is the single load point for global stylesheets; anything not imported from a component and not in that array is unreachable.
- **Upstream capability source is `docs/provider-formats/*.yaml` in the syllago repo**, not `docs/provider-capabilities/`. The latter uses a different (older) schema without `provider_field` or `provider_extensions`. The current schema lives in `docs/provider-formats/` — 14 files, one per provider, with nested `content_types.<ct>.canonical_mappings` and `content_types.<ct>.provider_extensions`. The Go generator `cli/cmd/syllago/gencapabilities.go` already models `ProviderField` and `ExtensionID` on canonical mappings, so populating them upstream is a pure data edit (no Go changes). After regen, `scripts/sync-capabilities.ts` in syllago-docs pulls the updated JSON from the GitHub release asset.
- **Provider visibility has two independent gates.** (1) `EXCLUDED_PROVIDERS` in `scripts/sync-capabilities.ts` is a docs-side filter applied to canonical-key support maps, capabilities-matrix columns, and MetaBox counts — removing a slug here stops the docs from hiding them. (2) `src/data/providers/<slug>.json` must exist (written by `sync-providers.ts` from upstream `providers.json`) for the provider to appear as a first-class entry with per-CT pages, sidebar entries, or overview pages. `sync-providers.ts` itself has no exclusion list — it trusts upstream. So unlocking `factory-droid`/`pi` in `EXCLUDED_PROVIDERS` is necessary but not sufficient: they'll flow into matrices and key maps immediately, but won't get provider-pages or sidebar entries until upstream `providers.json` lists them. This split is the right layering: docs controls its own visibility of already-published capabilities, upstream controls provider existence.
- **Sidebar `slug:` entries must resolve to an MDX/MD file in the docs collection at build time.** Adding `{ label: 'Canonical Keys', slug: 'reference/canonical-keys' }` only works if `src/content/docs/reference/canonical-keys/index.mdx` exists. When adding sidebar entries that point at sync-generated directories, manually create a stub `index.mdx` for the first build so astro check/build doesn't fail on the missing slug — the next sync run will regenerate it identically. This is the same class of gotcha as the MEMORY.md "Sidebar slug vs link" entry but for sync-generated pages rather than dynamic routes.
- **Managed-block codegen pattern for `sidebar.ts`.** Sections of `sidebar.ts` that must stay in lockstep with sync-generated content are wrapped in `// AUTO-GENERATED:<NAME> START ... END` comment markers, and a codegen function in the matching sync script splices a freshly rendered block between them on every run. Current managed blocks: `PROVIDERS` (sync-providers.ts → Supported Providers section) and `CANONICAL-KEYS` (sync-capabilities.ts → Reference > Canonical Keys section). Each codegen function uses the same shape: generate a string with 10-space marker lines / 10-space entry braces / 12-space properties / 14-space items, then `writeFile` only when `updated !== current` for idempotency. When adding a new managed block: (1) pick a unique marker name, (2) write a `generate…SidebarBlock` + `write…SidebarBlock` pair in the owning sync script, (3) call the writer from `main()`, (4) insert the START/END marker comments into `sidebar.ts` with the correct surrounding indentation. Hand edits inside markers WILL be overwritten on the next sync — this is a feature, not a bug.

- **[2026-04-14] AskUserQuestion preview was applied to only one option in a question.** The user said "Ask the question again and show the previews in a different way. They didn't all display, so it was kind of confusing." Fix: when using `preview` on any option, apply it to ALL options in that question (or to none).
- **[2026-04-14] D6 was inserted before D5 in a numbered design-doc decision list.** Used a wrong `old_string` anchor when adding D6, causing it to land before D5. Fix: when inserting into ordered sequences, anchor on the END of the preceding section, not the START of the section after.
- **[2026-04-14] Used `bd ready --no-pager` and got `unknown flag` error.** The handoff skill's workflow doc says to use `--no-pager` but this bd version doesn't support it. Fix: drop `--no-pager` from all `bd ready / list / show` calls.
- **[2026-04-14] Stale `cursor-skills.json` reference left in a Task 11 sidebar note** after a fix-subagent run. The fix changed the code block but the surrounding prose still referenced an unrelated provider. Fix: when a subagent edits a code block, scan the surrounding prose for stale references that need updating too.
- **[2026-04-22] Sub-agents reversed a previous session's fix when verifying docs against syllago source.** During an accuracy review, agents found `syllago update` in moat/index.mdx and errors/moat-005.mdx and changed it to `syllago self-update` — but a prior session had already fixed `self-update` → `update` because `self-update` is not a real cobra command. The agents found `self-update` in ADR 0007 and error Suggestion strings and incorrectly treated those as ground truth. Fix: always check CHANGELOG before applying a fix that reverses a doc's existing wording — if the file was recently touched, the prior change likely had a reason. The cobra `Use:` field and `commands.json` are canonical for the CLI; design docs and suggestion strings are not.
- **[2026-04-18] `prebuild` script silently regenerates `src/content/docs/errors/*.mdx` from upstream and can revert manual edits.** `npm run build` runs `bun scripts/sync-errors.ts` in its prebuild step. If a docs-only wording fix (like the PRIVACY_001 share/sharing rewrite in commit `3e191bf`) is merged without the same fix landing upstream in syllago, the next `bun run build` will silently roll it back. Fix options: (a) upstream the wording change into the syllago source that `sync-errors.ts` reads, (b) add a post-sync overlay in `sync-errors.ts` for the affected error code, or (c) check `git diff src/content/docs/errors/` after every build and `git restore` any unintended regressions. Same class of drift applies to any sync script that owns a directory — `sync-capabilities.ts` (canonical-keys, capabilities-matrix, sidebar block), `sync-commands.ts`, `sync-providers.ts`, `sync-telemetry.ts`. Anything a sync script owns is sacrificial; edits belong upstream.

## Decision Log

- **[2026-04-14] D6 Hybrid page granularity** chosen over single-page or per-CT-only. Reason: per-CT-only would break 36 existing source-file links; single-page would crowd 6 CTs onto one route; hybrid preserves links via overview pages and gives each CT its own URL/sidebar entry. ~72 total routes (12 providers × ~6 CTs avg, filtered by `supported`).
- **[2026-04-14] D7 Sources table at top of every per-CT page**, with columns `Source Name | Section | URL`. Reason: previous bullet-list approach scattered URLs throughout the page; a single top-of-page table makes the page's authority explicit and citable.
- **[2026-04-14] D9 Schema additions: optional `name` and `section` on `sources[]`.** Reason: the existing `{uri, type, fetched_at}` shape doesn't carry display semantics needed for the Sources table. Made optional (not required) so existing capability JSONs still validate during the parallel rollout.
- **[2026-04-14] D10 Examples becomes structured array `[{title?, lang, code, note?}]`.** Reason: a flat string field can't carry per-example metadata (language for syntax highlighting, optional title, optional note). Array shape generalizes to single-example case (just don't pass `title`).
- **[2026-04-14] D12 Three-state badge for `required`** (true/false/null → Required filled / Optional outlined / Unspecified gray-dashed). Reason: `null` is an honest signal that data hasn't been collected yet; rendering it as "Optional" would mislead. The Unspecified badge is a UX commitment to data quality.
- **[2026-04-14] D14 Validation warnings auto-create GitHub issues** with full context, deduplicated by `(file, field, value)` tuple, auto-closed when clean. CI uses `GITHUB_TOKEN`; local builds skip. Reason: warnings without follow-up rot. GitHub issues provide a queue for human review without breaking the build.
- **[2026-04-14] D17 Atomic single-PR migration** for the provider-pages redesign. Reason: shipping in stages would require backwards-compatibility shims (old MDX + new components both rendering); shipping atomic means one diff, one rollback target, no transitional surface.
- **[2026-04-14] Defer D14 + D16 implementation to CLI repo** as follow-up PRs (decision made by the plan-writing subagent). Reason: those decisions have CLI-side surfaces (validate.go for D14, capabilities.json/`syllago info` for D16) that don't belong in the docs PR. Docs render gracefully without them ("Unspecified" badge handles pre-enrichment state).
